title: Deploy Cookiecutter Django with Docker Compose
description: Run your Cookiecutter Django project in production using Docker Compose, Gunicorn, Traefik, and automatic HTTPS via Let's Encrypt.
---

The production Docker Compose stack for Cookiecutter Django is designed to be self-contained and ready to run on any Linux server with Docker installed. It wires together your Django application, a PostgreSQL database, Redis, and a Traefik reverse proxy that handles HTTPS automatically through Let's Encrypt — with no manual certificate management required.

## Prerequisites

- Docker 17.05 or higher
- Docker Compose 1.17 or higher

## Understanding the Docker Compose setup

Before you begin, review the `docker-compose.production.yml` file in the root of your project. It defines the following core services:

| Service | Role |
|---|---|
| `django` | Your application, served by Gunicorn |
| `postgres` | PostgreSQL database |
| `redis` | Redis instance for caching |
| `traefik` | Reverse proxy with automatic HTTPS |

If you opted into Celery (by setting `use_celery` to `y`), three additional services are included:

- `celeryworker` — runs a Celery worker process
- `celerybeat` — runs a Celery beat process
- `flower` — runs [Flower](https://github.com/mher/flower), served by Traefik over HTTPS on port `5555`

## Configuring the stack

Most services are configured through environment variables. Check your `.envs/.production/` directory for the relevant files.

### Sentry error logging

To capture errors in production, create an account on [sentry.io](https://sentry.io/welcome) and set the `SENTRY_DSN` environment variable. Log entries at `logging.ERROR` level and above are forwarded to Sentry as events:

```python
import logging
logging.error("This event is sent to Sentry", extra={"<example_key>": "<example_value>"})
```

The `extra` parameter lets you attach additional context to each event.

### Mail backend

You need to configure a mail backend before users can register or log in. Without it, the account creation view will raise a 500 error when it tries to send a verification email. Add your mail provider's API key and sender domain to `.envs/.production/.django`.

<Warning>
  If your email server is not configured properly (Mailgun by default), attempting to send an email will cause an Internal Server Error.

  By default, `django-allauth` is set up to require email verification, which means it sends a verification email when an unverified user tries to log in or when someone signs up.

  This can happen immediately after setting up a Mailgun account, which runs in a sandbox subdomain by default. Either add your email to the list of authorized recipients or verify your domain.
</Warning>

## Optional: Use AWS IAM role for EC2

If you are deploying to AWS, you can use an IAM role instead of static credentials. Once configured, it is safe to remove `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` from `.envs/.production/.django`.

To set this up:

1. Create an [IAM role](https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/iam-roles-for-amazon-ec2.html) with at minimum the `AmazonS3FullAccess` permission.
2. [Attach the role](https://aws.amazon.com/blogs/security/easily-replace-or-attach-an-iam-role-to-an-existing-ec2-instance-by-using-the-ec2-console/) to your existing EC2 instance, or launch a new instance with the role assigned.

## HTTPS is on by default

Traefik automatically obtains and renews a valid TLS certificate from Let's Encrypt for your domain. All you need to do is ensure your DNS records point to the server where Traefik is running.

Two things to keep in mind:

- Set your server's IP address in the `DJANGO_ALLOWED_HOSTS` environment variable before deploying. Without it, the site will be unreachable over HTTP.
- Access to the Django admin is restricted to HTTPS in production.

You can read more about the Traefik configuration in the [Automatic HTTPS](https://docs.traefik.io/https/acme/) documentation.

## Webpack without Whitenoise

If you chose Webpack without Whitenoise, Webpack needs to know the static file URL at build time — specifically when `docker compose build` runs (see `webpack/prod.config.js`). The relevant variables are:

- `AWS_STORAGE_BUCKET_NAME`
- `DJANGO_AWS_S3_CUSTOM_DOMAIN`
- `DJANGO_GCP_STORAGE_BUCKET_NAME`
- `DJANGO_AZURE_CONTAINER_NAME`

Django reads these from `.envs/.production/.django` at runtime, but Docker only reads a `.env` file in the project root at build time. If these values are missing during the build, the deployed site will have no CSS or JavaScript.

There are three ways to address this:

<AccordionGroup>
  <Accordion title="Option 1: Merge all env files into .env">
    Run the provided helper script to combine your production env files into a single `.env` at the project root:

    ```bash
    python merge_production_dotenvs_in_dotenv.py
    ```
  </Accordion>
  <Accordion title="Option 2: Create a minimal .env file">
    Create a `.env` in the project root containing only the variables Webpack needs. You will also need to keep them in `.envs/.production/.django`, so they will be duplicated.
  </Accordion>
  <Accordion title="Option 3: Set variables inline at build time">
    Pass the variables directly when running the build command:

    ```bash
    DJANGO_AWS_S3_CUSTOM_DOMAIN=example.com docker compose -f docker-compose.production.yml build
    ```
  </Accordion>
</AccordionGroup>

## Building and running the production stack

<Steps>
  <Step title="Build the stack">
    ```bash
    docker compose -f docker-compose.production.yml build
    ```
  </Step>
  <Step title="Start the stack">
    Run attached to see logs in the terminal:

    ```bash
    docker compose -f docker-compose.production.yml up
    ```

    Or run detached in the background:

    ```bash
    docker compose -f docker-compose.production.yml up -d
    ```
  </Step>
  <Step title="Run database migrations">
    Open a second terminal and run:

    ```bash
    docker compose -f docker-compose.production.yml run --rm django python manage.py migrate
    ```
  </Step>
  <Step title="Create a superuser">
    ```bash
    docker compose -f docker-compose.production.yml run --rm django python manage.py createsuperuser
    ```
  </Step>
</Steps>

### Other useful commands

Check the logs:

```bash
docker compose -f docker-compose.production.yml logs
```

Open a Django shell:

```bash
docker compose -f docker-compose.production.yml run --rm django python manage.py shell
```

Scale the application horizontally:

```bash
docker compose -f docker-compose.production.yml up --scale django=4
docker compose -f docker-compose.production.yml up --scale celeryworker=2
```

<Warning>
  Do not scale the `postgres`, `celerybeat`, or `traefik` services. Running multiple instances of these will cause data corruption or unexpected behavior.
</Warning>

Check container status:

```bash
docker compose -f docker-compose.production.yml ps
```

## Process management with Supervisor

To ensure the stack restarts automatically after a reboot or crash, run it under a process manager. Any process manager works — it just needs to execute `docker compose -f docker-compose.production.yml up` in your project's root directory.

Here is a starting-point configuration for [Supervisor](http://supervisord.org/):

```ini
[program:my_project]
command=docker compose -f docker-compose.production.yml up
directory=/path/to/my_project
redirect_stderr=true
autostart=true
autorestart=true
priority=10
```

Save this file to `/etc/supervisor/conf.d/my_project.conf`, then activate it:

```bash
supervisorctl reread
supervisorctl update
supervisorctl start my_project
```

Check status at any time with:

```bash
supervisorctl status
```

## Media files without a cloud provider

If you chose no cloud provider, media files are served by an nginx service from the `production_django_media` Docker volume. Make sure this volume persists across deployments — it is not backed up automatically, and deleting it will permanently remove uploaded files.
